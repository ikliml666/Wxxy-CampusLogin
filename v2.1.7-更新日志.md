# v2.1.7 更新日志

## 🚀 新功能

### DNS智能解析 — 应用自主选择DNS服务器 + DoH回退

#### 功能描述

网络质量分析中的DNS解析模块不再依赖系统DoH API，改为应用程序自主选择最优DNS服务器并发起连接。当传统DNS解析失败时，自动回退到应用级DoH解析，彻底避免系统DoH API导致的DNS解析问题。

#### 技术实现

**DNS服务器动态评分系统（`http_timing.rs`）：**
- 新增 `DNS_SERVER_SCORES` / `DOH_SERVER_SCORES` 全局状态表，跟踪每个DNS/DoH服务器的延迟和可用性
- `get_best_dns_servers()` 按延迟升序返回可用服务器，600秒过期自动回退默认列表
- `get_best_doh_servers()` 同理，并关联对应IP地址
- 网络质量检测中DNS/DoH测试结果实时反馈到评分表，形成闭环

**应用级DoH解析（`resolve_via_doh`）：**
- 直接TCP连接DoH服务器443端口 → TLS握手 → 发送 `GET /dns-query?name=xxx&type=A`
- 解析HTTP响应体JSON，提取A记录IP地址
- 完全绕过系统DoH API，由应用程序自己发起对DNS服务器的连接

**三级智能解析策略（`resolve_host_smart`）：**
```
DNS缓存 → 传统DNS(动态选最优服务器) → DoH回退(按延迟排序尝试)
```
- 第一级：查询DNS缓存（TTL 60秒）
- 第二级：传统DNS解析，使用延迟最优的服务器并发查询（`num_concurrent_reqs` 动态调整）
- 第三级：DoH回退，按延迟排序依次尝试各DoH服务器

**两阶段网络质量检测（`quality.rs`）：**
- Phase 1：并行测试网关 + DNS服务器 + DoH服务器 → 更新评分表
- Phase 2：并行预解析HTTPS主机名 → 并行测试HTTPS网站
- HTTPS测试使用Phase 1更新的最优DNS服务器，解析成功率更高

#### 删除的旧机制

| 删除项 | 原因 |
|--------|------|
| `DnsResolve` 任务类型 | 仅测试单个硬编码DNS服务器，与 `DnsServer` 任务重复 |
| `check_dns_latency_async` 函数 | 仅被 `DnsResolve` 使用，删除后无引用 |
| `prefetch_dns_cache` 函数 | 无外部调用者，`resolve_host_smart` 已内置缓存逻辑 |
| `resolve_host_uncached` 硬编码DNS服务器 | 改为动态从评分表获取最优服务器 |

### 通知系统优化 — 统一在线状态通知

#### 问题描述
启动时已在线的场景下，三个独立系统（自动登录、后台检测、适配器监控）同时检测到"已在线"状态，各自发出通知，导致日志中出现 4 条重复信息：
```
07:57:34 已在线
07:57:34 以太网: 已在线
07:57:34 WLAN: 已在线
07:57:49 网络适配器状态已更新
```

#### 修改内容

**后端 `auto_login.rs`：**
- 启动自动登录检测到已在线时，设置 `was_online` 和 `has_logged_online` 标志，防止后台检测重复报告状态变更
- 改善消息格式：从 `"已在线, 已在线"` 改为 `"已在线（以太网、WLAN）"`，包含适配器名称

**前端 `useAppInit.ts`：**
- `onBackgroundCheckResult`：主/副适配器状态变更合并为一条日志，格式统一为 `"已在线（以太网、WLAN）"` / `"以太网: 已离线"`
- `onBackgroundCheckResult`：添加 5 秒内在线日志去重机制，自动登录已报告在线后，后台检测的重复在线日志自动跳过
- `onAutoLoginResult` skipped：移除 `"已在线，跳过登录: "` 冗余前缀，直接显示后端格式化的消息，类型从 `info` 改为 `success`
- `onAdaptersChanged`：移除 `"网络适配器状态已更新"` 日志（UI 已实时反映适配器变化）
- `onAdaptersChanged`：不再清空 `lastAdapterOnlineRef`，避免适配器变更后后台检测重复报告在线状态

**优化后启动日志（已在线场景）：**
```
07:57:34 已在线（以太网、WLAN）
```

### 延迟胶囊背景颜色修复

#### 问题描述
状态栏中内网延迟、外网延迟、DNS解析胶囊的背景颜色消失。

#### 根因分析
`getLatencyColor()` 函数通过 `(cfg.border).replace('border-', 'bg-')` 动态生成 `bg-*-500/20` 类名（如 `border-emerald-500/20` → `bg-emerald-500/20`）。Tailwind JIT 编译器无法扫描到动态生成的类名，这些类之前恰好出现在 DashboardPanel.tsx 的 mini 变体代码中而被"意外"包含。移除 mini 变体后，这些 CSS 类不再被生成。

#### 修改内容

- `QUALITY_CONFIG`（`constants/index.ts`）：新增显式 `borderBg` 字段，直接声明 `bg-emerald-500/20` 等类名
- `getLatencyColor()`（`lib/latency.ts`）：使用 `cfg.borderBg` 替代 `(cfg.border).replace('border-', 'bg-')` 动态字符串替换

## 🔍 缺陷分析与修复

### 适配器缓存机制修复（B1+P3）

#### 问题描述
`get_adapters_force()` 函数名为"强制刷新"，但实际直接调用 `get_adapters_cached()`，无法获取最新适配器数据。`wait_for_adapter()` 和 `run_auto_login_on_start()` 依赖此函数获取最新适配器列表，可能拿到过期数据。同时，每次调用 `get_adapters_cached()` 都执行 `GetAdaptersAddresses` Win32 API，频繁调用系统API。

#### 修改内容

**后端 `adapter.rs`：**
- 引入 `ADAPTER_CACHE` 全局缓存（`Mutex<Option<(Vec<Adapter>, Vec<AdapterDetail>, Vec<DisabledAdapter>, Instant)>>`），TTL 为 5 秒
- `get_adapters_cached()` / `get_disabled_adapters_cached()` / `get_adapter_details_cached()` 优先读取缓存，5秒内复用
- `get_adapters_force()` / `get_disabled_adapters_force()` 先清除缓存再查询，实现真正的强制刷新
- 提取 `query_adapters_cached_inner()` 统一缓存读写逻辑

---

### 登录URL构造逻辑修复（B2）

#### 问题描述
Portal登录URL的 `:801` 端口追加逻辑使用 `contains(":801/")` 判断，存在两个问题：
1. 若用户配置的 `portalUrl` 已含 `:801` 但后无 `/`，会重复追加 `:801`
2. 若用户配置了自定义端口（如 `:8080`），也会被错误追加 `:801`，生成非法URL如 `http://10.1.99.100:8080:801/eportal/portal/login`

#### 修改内容

**后端 `login_request.rs`：**
- 改用 `contains(":801")` 判断（不含尾部 `/`），避免已含端口时重复追加
- 先 `trim_end_matches('/')` 去除尾部斜杠再拼接

**后端 `portal.rs`：**
- 同步修复 `check_portal_full` 中的 `:801` 追加逻辑
- 提取 `portal_base_with_port` 变量，统一端口处理

### 配置保存重试机制（B8）

#### 问题描述
`atomic_write` 在 Windows 上使用 `fs::rename` 替换配置文件，当目标文件被其他进程（如杀毒软件）短暂锁定时会直接失败，无重试机制。

#### 修改内容

**后端 `config.rs`：**
- `atomic_write` 添加 3 次重试，每次间隔 100ms
- 重试全部失败后才清理临时文件并返回错误

---

### 前端密码处理简化（B9）

#### 问题描述
`passwordSaved` / `passwordExplicitlyCleared` / `***` 哨兵值三者交织，存在风险路径：用户输入密码→保存→再次打开设置→密码显示 `***`→用户不修改直接保存→`***` 被当作新密码。后端虽有兜底，但前端逻辑本身复杂且脆弱。

#### 修改内容

**前端 `useAppStore.ts`：**
- 移除 `passwordExplicitlyCleared` 状态变量
- `updateConfig`：当 `password === '***'` 时直接 `delete sanitized.password`，不再发送密码字段
- `saveConfigDirect`：简化为 `passwordSaved && !saveData.password` 时发送 `***`（告知后端保留原密码）
- `doLogin`：同步简化密码判断条件

## 🚀 新功能

### DNS优化 — 检测与一键设置推荐DNS+DoH

#### 功能描述

在网络面板新增"DNS优化"卡片，帮助用户检测当前DNS配置，并一键设置推荐的阿里DNS(223.5.5.5)和腾讯DNS(119.29.29.29)，同时启用DNS over HTTPS (DoH)加密。

#### 功能特性

- **DNS状态检测**：读取Windows注册表获取每个活动适配器的DNS服务器地址和DoH状态
- **DoH支持检测**：自动检测系统是否支持DoH配置（Windows 11 / Windows Server 2022+）
- **一键优化**：设置推荐DNS服务器 + 启用DoH加密，无需手动操作注册表
- **状态分级**：DNS+DoH（优秀）/ DNS（良好）/ 待优化 / 未配置，直观展示当前配置质量

#### 技术实现

**后端（注册表API替代PowerShell）：**
- `check_dns_doh_status`：通过 `winreg` crate 直接读取Windows注册表，检测速度比PowerShell快10倍以上
  - 适配器DNS：`HKLM\SYSTEM\CurrentControlSet\Services\Tcpip\Parameters\Interfaces\{GUID}\NameServer`
  - 适配器名称映射：`HKLM\SYSTEM\CurrentControlSet\Control\Network\{4D36E972-...}\{GUID}\Connection\Name`
  - DoH配置：`HKLM\SYSTEM\CurrentControlSet\Services\Dnscache\Parameters\DohWellKnownServers\{IP}`
- `setup_recommended_dns`：通过 `winreg` crate 直接写入注册表设置DNS和DoH，避免PowerShell进程启动开销

**前端：**
- `types/index.ts`：新增 `DnsDohStatus`、`DnsAdapterInfo`、`DnsServerInfo` 类型
- `useIpc.ts`：新增 `checkDnsDohStatus`、`setupRecommendedDns` API
- `NetworkPanel.tsx`：新增"DNS优化"卡片，支持检测、状态展示、一键设置

### 校园网注销功能

#### 功能描述

新增校园网注销/下线功能，用户可以在应用内直接注销校园网登录，无需再通过浏览器打开认证门户手动操作。

#### 技术实现

**注销 API：**
调用校园网 Portal 的 MAC 解绑端点实现注销：
```
GET http://10.1.99.100:801/eportal/portal/mac/unbind?callback=dr1002&user_account={学号}&wlan_user_mac=&wlan_user_ip=&jsVersion=4.1.3&v=526&lang=zh
```

与登录 API 的关键差异：
- 端点为 `/eportal/portal/mac/unbind`（登录为 `/login`）
- `user_account` 只传纯学号，不带 `@运营商` 后缀
- 不需要密码参数
- JSONP 回调为 `dr1002`（登录为 `dr1003`）

**后端：**
- `login_request.rs`：新增 `do_logout_request()`、`do_logout_with_retry()`、`parse_logout_result()` 函数
- `login.rs`：新增 `logout_adapter_with_log()`、`full_logout_inner()`、`do_logout` Tauri 命令
- `do_logout` 和 `do_login` 均支持可选 `adapter_name` 参数，指定适配器时只操作该适配器，不指定时按配置执行全部适配器
- 双适配器模式下串行注销（与登录逻辑一致）
- 注销重试最多 2 次，重试等待中检查退出标志

**前端：**
- `useIpc.ts`：`doLogin` / `doLogout` 均新增 `adapterName?: string` 可选参数
- `useAppStore.ts`：新增 `isLoggingOut` 状态和 `doLogout` action，日志中显示注销目标
- `DockNav.tsx`：新增注销按钮，与登录按钮并列

**注销成功判定：**
- `msg` 包含 "解绑终端MAC成功" → 注销成功
- `msg` 包含 "获取用户在线信息数据为空" → 当前无在线设备（也视为成功）

## 🔍 深度缺陷分析修复

### 注销锁语义修复

#### 问题描述
`do_logout` 复用了 `is_logging_in` 锁，导致注销时用户收到"操作正在进行中"的提示，无法理解发生了什么。登录和注销是两种不同操作，应使用独立的互斥锁。

#### 修改内容

**后端 `state.rs`：**
- `TaskFlags` 新增 `is_logging_out: TaskLock` 字段

**后端 `login.rs`：**
- `do_logout` 使用 `is_logging_out` 替代 `is_logging_in`
- 错误提示从"操作正在进行中"改为"注销正在进行中"

---

### 配置保存密码保护增强

#### 问题描述
前端 `saveConfigDirect` 在 `passwordSaved=true && password为空` 时发送 `"***"` 占位符，后端将 `"***"` 还原为旧密码。但当用户修改其他配置（如切换主题）时，`updateConfig` 过滤掉 `"***"` 后前端 `delete saveData.password`，Tauri 反序列化默认为空字符串 `""`，后端收到空密码会覆盖旧密码，导致保存的密码被意外清空。

#### 修改内容

**前端 `useAppStore.ts`：**
- `saveConfigDirect` 和 `doLogin` 中 `delete saveData.password` 替代 `saveData.password = '***'`，不再发送 `"***"` 占位符

**后端 `config_cmd.rs`：**
- `save_config` 增加 `password.is_empty() && !guard.password.is_empty()` 判断：当前端未传密码（反序列化为空字符串）且旧密码存在时，保留旧密码
- 完整逻辑：`"***"` → 保留旧密码；空字符串+旧密码存在 → 保留旧密码；空字符串+旧密码为空 → 无密码；实际密码 → 使用新密码

---

### 移除网络质量检测中的DNS预取

#### 问题描述
`check_network_quality_async` 在 HTTPS 延迟测试前，先并发预解析 12 个域名的 DNS，目的是让后续测试命中缓存。但 `measure_https_timing` 已单独测量 `dns_ms`，DNS 解析时间不会混入 HTTPS 延迟，预取对结果准确性无帮助。12 个并发 DNS 查询白白消耗网络资源，属于"为了性能而性能"。

#### 修改内容

**后端 `quality.rs`：**
- 删除 `prefetch_set` 相关代码（JoinSet 创建、spawn、join_next 循环）
- HTTPS 测试自然触发 DNS 解析，DNS 缓存机制已能避免重复解析

---

### 配置写入失败时保留临时文件

#### 问题描述
`atomic_write` 重试3次失败后执行 `std::fs::remove_file(&tmp_path)` 删除临时文件，导致配置数据丢失且无法恢复。临时文件包含最新的配置内容，应保留以便用户手动恢复。

#### 修改内容

**后端 `config.rs`：**
- 删除 `let _ = std::fs::remove_file(&tmp_path)`
- 错误信息附带临时文件路径：`format!("重命名临时文件失败（重试3次后），临时文件保留在: {:?}", tmp_path)`

## 🐛 Bug 修复（v2.1.7 追加）

### DNS解析显示"--"修复

#### 问题描述
网络质量面板中，阿里DoH、腾讯DoH、阿里DNS、腾讯DNS、信风DNS、DNS解析等测试项全部显示"--"，顶部状态栏DNS胶囊持续转圈。

#### 根因分析

1. **DoH请求格式错误**：`resolve_via_doh` 和 `do_doh_https` 使用 `?name=baidu.com&type=A` + `Accept: application/dns-json`（Google JSON API格式），但阿里DNS和腾讯DNS的 `/dns-query` 端点仅支持 RFC 8484 wire format（`?dns=<base64url>` + `Accept: application/dns-message`），导致所有DoH请求返回错误
2. **"DNS解析"测试项缺失**：前端 QualityPanel 和 StatusBar 引用了 `details['DNS解析'].latency`，但后端从未提供该数据，导致胶囊永远转圈
3. **DNS查询超时太短**：2000ms超时 + 仅1次重试，校园网环境下容易失败

#### 修改内容

**后端 `http_timing.rs`：**
- 新增 `build_dns_query_wire()`：构建标准DNS wire format查询报文
- 新增 `base64url_encode_no_pad()`：RFC 8484要求的base64url编码（无填充）
- 新增 `parse_dns_response_wire()`：解析DNS wire format响应，提取A记录IP
- 重写 `resolve_via_doh`：改用 `?dns=<base64url>` + `Accept: application/dns-message`
- 修复 `do_doh_https`：同步改用 RFC 8484 wire format
- `resolve_host_uncached_with_bind`：自定义DNS失败时自动回退到系统DNS（`ResolverConfig::default()`）
- DNS查询超时从2000ms增加到3000ms，重试次数从1增加到2，启用 `try_tcp_on_error`

**后端 `quality.rs`：**
- 新增 `SystemDns` 任务类型：测量系统DNS解析延迟
- 添加"DNS解析"测试项：解析 `www.baidu.com`、`www.bilibili.com`、`www.jd.com`、`cn.bing.com` 四个网站取平均延迟
- DNS查询超时从2000ms增加到3000ms

### DNS优化按钮消失修复

#### 问题描述
网络面板中"一键优化DNS"按钮不显示。

#### 根因分析
`should_filter_ip` 函数过滤了所有私有IP地址（10.x.x.x、172.16-31.x.x、192.168.x.x），校园网DNS服务器通常在10.x网段，过滤后适配器DNS列表为空，按钮条件判断返回false。

#### 修改内容

**后端 `network_cmd.rs`：**
- `should_filter_ip` 不再过滤私有IP，仅过滤网络地址(.0)、广播地址(.255)、回环(127.x)、APIPA(169.254.x)、代理假DNS(198.18-19.x)
- `netsh dns show encryption` 解析改用 `find()` 在整行搜索IP，兼容中英文Windows输出
- autoupgrade检测兼容中文"是"和英文"yes"
- 设置DNS后自动执行 `ipconfig /flushdns`

**前端 `NetworkPanel.tsx`：**
- "一键优化DNS"按钮改为始终显示，不再依赖先检测DNS

### PowerShell依赖消除 — UAC提权改用ShellExecuteW

#### 优化描述

#### 修改内容

- 新增 `run_elevated()` 函数：使用 `ShellExecuteW` + `"runas"` 实现UAC提权

### setup_dns_doh 重写 — 零PowerShell实现

#### 优化描述
一键优化DNS功能完全重写，所有操作使用netsh原生命令 + ShellExecuteW提权，消除PowerShell依赖。

| 操作 | 之前 | 现在 |
|------|------|------|
| 获取适配器 | PowerShell `Get-NetAdapter` | `get_adapters_cached()` (Win32 API) |

| 注册DoH | PowerShell `Add-DnsClientDohServerAddress` | `netsh dns add encryption` |
| UAC提权 | PowerShell `Start-Process -Verb RunAs` | `ShellExecuteW` Win32 API |
| DNS缓存刷新 | PowerShell脚本内 | `ipconfig /flushdns` |
| 整体耗时 | ~1-3s | ~100-300ms |
